### Technical Architecture: Building Multi-Agent Conversations with AWS Services and Amazon Bedrock

So, yeah, let's move on to  the explanation about the architecture. Here's the overall architecture of the application. So not only do we deploy agents for each appliance avatar, but we also have an agent managing the memory system and the memory calendar. The front end uses React Router, GSAP, and Pixi JS, all of which are built by Amazon Q Developer, currently unified under the name of Amazon Q. Conversations between users and the avatars are managed in DynamoDB and connected to the front end via GraphQL through AppSync. Additionally, since we often use a serverless architecture, the back end is primarily built around Lambda and Step Functions.

[![Thumbnail 430](https://raw.githubusercontent.com/kazuya-iwami/devto-repo/main/posts/images/7098fa8b4cdd4398/430.jpg)](https://www.youtube.com/watch?v=ab7FuwuZLz0&t=430)

So we have four features  in this architecture. I would like to explain each feature. The first one is the unique personality and features of each appliance avatar. Each appliance agent includes a system prompt and a personal profile that differentiates its voice and behavior. We also maintain an allowed list so agents can call only approved device APIs. Personas can be tuned for dialect and topic preferences, and we use Llama's template and safety constraints to keep replies safe and appropriate. Agents can invoke external APIs like switchboard APIs in this case to perform diverse actions and retrieve context-aware information such as energy use or recipe suggestions.

[![Thumbnail 490](https://raw.githubusercontent.com/kazuya-iwami/devto-repo/main/posts/images/7098fa8b4cdd4398/490.jpg)](https://www.youtube.com/watch?v=ab7FuwuZLz0&t=490)

 The second feature is a personalized knowledge base. We use a structured knowledge model using a graph database via Amazon Neptune. Nodes represent utterances and preferences, while edges capture relationships and temporal context. We combine rule-based heuristics with learned models to assess the importance of information and effectively store only high-impact memories from every conversation. By analyzing conversation flow, agents gradually learn context and improve future replies.

[![Thumbnail 540](https://raw.githubusercontent.com/kazuya-iwami/devto-repo/main/posts/images/7098fa8b4cdd4398/540.jpg)](https://www.youtube.com/watch?v=ab7FuwuZLz0&t=540)

The third feature is a memory calendar.  In the demo, in the last of the demos I played, the video demo, that one is the memory calendar. Conversations are preserved in layered forms such as text summaries, images such as thumbnails for memorable moments, and short video clips when relevant. This creates story-like memories that you can browse like a picture diary. We also classify topics so the agent can retrieve relevant past interactions and enrich future replies.

[![Thumbnail 580](https://raw.githubusercontent.com/kazuya-iwami/devto-repo/main/posts/images/7098fa8b4cdd4398/580.jpg)](https://www.youtube.com/watch?v=ab7FuwuZLz0&t=580)

 The last feature is a conversation mediator. We use a mediator to keep the multi-avatar conversation smooth and comfortable. The mediator scores candidate replies for relevance, confidence, and personal fit. It enforces constraints such as one utterance per avatar to prevent one avatar from dominating the conversation.

[![Thumbnail 620](https://raw.githubusercontent.com/kazuya-iwami/devto-repo/main/posts/images/7098fa8b4cdd4398/620.jpg)](https://www.youtube.com/watch?v=ab7FuwuZLz0&t=620)

The merit is the natural turn-taking and the very rapid crosstalk or flip-flop, so interaction feels calm and human.  I think maybe you're wondering why we're not using Amazon Bedrock Agent Core. At that time, we couldn't adopt Amazon Bedrock Agent Core because it was still pre-release when we developed it. However, many features could likely be replaced by Amazon Bedrock Agent Core, I think.

Agent Core runtime runs AI agent frameworks like LangChain agents or Agent Core on Amazon Bedrock LLMs, which speeds up agent hosting and development. Agent Core gateway covers the existing APIs and functions into MCP-compatible tools so agents can easily call diverse functions. Memory, identical memory, manages the memory access for short-term and long-term stores, providing agents with the right content at the right time.
